Unsigned Libraries
Venus Library Manager » Unsigned Libraries

Overview

The Unsigned Libraries feature provides a convenience mechanism for discovering and working with .hsl and .smt files in the Library folder that were not installed through Venus Library Manager and are not part of any Hamilton system library. These are files that were placed in the Library folder manually, by another tool, or by an older VENUS installation - they exist on disk but have no corresponding database record or signed package.

This feature is purely for convenience. It performs no hashing, integrity checking, or validation of unsigned files. Its purpose is to make these libraries visible within the application so users can:

  • See what unmanaged libraries are present in their Library folder
  • Annotate them with metadata (author, version, description, tags)
  • Quickly export them as signed .hxlibpkg packages for proper distribution
  • Satisfy dependency resolution - unsigned libraries can fulfill #include dependencies with a warning

Enabling the Feature

Unsigned library scanning is disabled by default. To enable it:

  1. Open Settings (gear icon in the navigation bar)
  2. Locate the Unsigned Libraries section
  3. Check the Include existing unsigned libraries on the front page checkbox
  4. Click the Scan Now button to perform the initial scan

The setting is persisted in settings.json under the key chk_includeUnsignedLibs. The Scan Now button is disabled when the checkbox is unchecked.

Scan on Launch

Below the Scan Now button, a Scan for new unsigned libraries each launch checkbox is available. When enabled, the application automatically performs an unsigned library scan every time it starts, so newly added files are detected without requiring a manual scan. This setting is persisted under the key chk_scanUnsignedOnLaunch and is only available when the main unsigned libraries checkbox is enabled.

Scanning

When you click Scan Now, the application performs the following steps:

  1. Reads the Library folder - recursively scans the directory configured via the Library Path setting (default: C:\Program Files (x86)\HAMILTON\Library). Files and directories that match the exclusion rules described below are skipped (see Excluded Files and Directories).
  2. Identifies primary files - discovers all .hsl and .smt files.
  3. Excludes claimed files - filters out any file that is already tracked by a system library (from system_libraries.json) or an installed user library (from installed_libs.json).
  4. Groups by library name - derives a library name from the filename (without extension). Language-variant suffixes (e.g., Enu, Deu, Jpn) are folded into the base library name.
  5. Discovers related files - for each library name, gathers associated files with related extensions: .hs_, .sub, .bmp, .ico, .chm, .stp, .res, .fdb, .sii, .dec, .dll.
  6. Resolves icons - if a .bmp file matching the library name is found, it is encoded as base64 and stored for display as the library card icon.
  7. Preserves metadata - if the library was discovered by a previous scan and the user has edited metadata (author, version, description, etc.), those edits are preserved across rescans.
  8. Saves results - all discovered unsigned libraries are stored in unsigned_libs.json in the user data directory.

Excluded Files and Directories

The scanner applies several exclusion rules to avoid picking up temporary files, hidden artifacts, and internal application directories. The following items are always skipped and will never appear as unsigned libraries in the Library Manager:

RuleApplies ToExamplesReason
Name starts with . (dot) Files and directories .MyLibrary.hsl, .tmp, .backup/ Dot-prefixed entries are temporary or hidden artifacts created by VENUS, text editors, version control systems, and other tools. They are not real libraries and should never be visible to users.
Name starts with ~ (tilde) Files only ~MyLibrary.hsl, ~$tempfile.smt Tilde-prefixed files are temporary lock/swap files created by editors and VENUS during saves.
Reserved directory names Directories only VenusLibraryManager, LibraryPackages, .VenusLibraryManager, LibraryIntegrityAudit Internal directories used by the Library Manager application itself. Scanning these would produce false positives.

Note: These exclusion rules are applied during the recursive directory scan. If a directory is excluded, none of its contents - including any subdirectories - are scanned. For example, a folder named .archive containing valid .hsl files will be entirely ignored.

Visual Feedback

While a scan is in progress, a spinning circle indicator appears next to the Scan Now button to show that work is being done. When the scan completes successfully, the spinner is replaced by a green checkmark icon confirming completion. A status message also shows how many unsigned libraries were found (e.g., "12 unsigned libraries found"). If the scan fails, both the spinner and checkmark are hidden and an error message is displayed instead.

The Unsigned Group

When the feature is enabled and at least one unsigned library has been discovered, a special Unsigned group appears in the left-side navigation bar. This group:

  • Uses the far fa-times-circle icon to visually distinguish it from other groups
  • Displays an "Auto-Detected" badge in the Settings accordion
  • Is locked - libraries cannot be dragged into or out of the Unsigned group
  • Is excluded from the sortable group ordering in Settings
  • Is not stored in tree.json - it is generated dynamically based on the scan results
  • Is hidden automatically when the feature is disabled or no unsigned libraries exist

Library Cards

Unsigned libraries appear as cards similar to system library cards, with the following differences:

  • A grey left border (instead of the system blue) indicates unsigned status
  • An "Unsigned" badge with the fa-times-circle icon is shown next to the library name
  • If a .bmp icon file was discovered, it is displayed; otherwise, the fa-times-circle placeholder icon is shown
  • The card shows the file count, version (if set), and author (if set)
  • An "Edit & Export" link opens the detail modal for editing metadata and exporting

Detail & Edit Modal

Clicking a library card name or the Edit & Export link opens the Unsigned Library Detail modal. This modal uses the same card-based layout as the Create Package form, providing a consistent visual experience across the application. Each section - Package Metadata, Library Icon / Image, Library Files, and Demo Method Files - is presented within its own styled card with a header, matching the packager form 1:1.

Editable Fields

The Package Metadata card contains the following form fields:

FieldDescription
Library NameDisplay name (pre-filled from the filename, read-only display in the modal header)
AuthorAuthor name - blank by default, fill in before exporting. The name Hamilton is restricted and requires password authorization (same restriction as the package creator form).
OrganizationAuthor's organization. The name Hamilton is restricted (same as Author).
VersionVersion string (e.g., 1.0.0)
VENUS CompatibilityMinimum VENUS version required
DescriptionFree-text description of the library
TagsComma-separated keywords for search and filtering. Tags are normalized to lowercase, spaces are removed, and invalid characters are stripped. Reserved/system words are blocked (see Tag Rules below).

Tag Rules (Limits and Exceptions)

  • Allowed characters: a-z, 0-9, and -.
  • Normalization: Tags are lowercased, repeated hyphens are collapsed, and leading/trailing hyphens are removed.
  • Length: Minimum 2 characters, maximum 24 characters.
  • Count: Up to 12 tags per library.
  • Numeric-only tags: Blocked.
  • Underscore exception: Underscores are blocked except for the explicit allowed tag ml_star.
  • Restricted/reserved words: Tags containing any reserved word are blocked. Reserved words include system, hamilton, read-only, stared, starred, signed, unsigned, registered, and unregistered.

If a tag is adjusted or blocked, the UI provides immediate feedback so users can correct input before saving/exporting.

Library Icon / Image

The Library Icon / Image card lets users assign a custom icon or image to the unsigned library. Supported formats include .png, .jpg, .jpeg, .bmp, .ico, .gif, and .svg. If no image is selected, the auto-detected .bmp file (if any) is used. A 72×72 pixel preview is displayed alongside Choose Image... and Remove buttons. The chosen image is also shown in the modal header next to the library name. When saved, the image data (library_image, library_image_base64, library_image_mime) is persisted to the database and included in exported packages.

Library Files

The Library Files card displays all files discovered for this library and allows adding more. Its card header includes a subtitle showing the install destination path: ...\Hamilton\Library\<libraryname>. Users can manage files using:

  • Add Files... - opens a file picker to add individual library files
  • Add Folder... - opens a folder picker; all files in the selected folder are added
  • Remove Selected - removes files selected in the list (click to select, Ctrl+click for multi-select)

A file count is displayed next to the buttons. Files are shown in a styled scrollable list container (matching the Create Package form). Files added by the user are persisted as additional_library_files in the database record, separate from the originally discovered files.

COM Registration

For any .dll file in the library file list, a COM Register checkbox is shown next to the filename. Check this box if the DLL requires COM registration via RegAsm.exe /codebase during import (administrator rights will be required).

Note: When one or more DLLs are marked for COM registration, a warning is displayed in the modal footer: "One or more DLLs are marked for COM registration. Administrator rights will be required during import." The package can still be exported normally - the COM registration data is included in the package manifest and will be applied during import.

When exactly one DLL is added in a single batch (via Add Files or Add Folder), it is automatically checked for COM registration. Uncheck it manually if registration is not needed.

Demo Method Files

The Demo Method Files card allows users to attach demo or example method files to the unsigned library. Its card header includes a subtitle showing the install destination path: ...\Hamilton\Methods\Library Demo Methods\<libraryname>. These files are placed in the demo_methods/ directory inside the exported package and installed to Hamilton\Methods\Library Demo Methods\<libraryname> during import.

  • Add Files... - opens a file picker for demo method files (.hsl, .stp, .csv, .txt, .med, .wfl)
  • Add Folder... - adds all files from a selected folder
  • Remove Selected - removes selected demo files from the list

Actions

  • Save - persists any metadata edits to the database. A visual confirmation ("Saved!") is shown briefly.
  • Remove - removes the library from the unsigned libraries list (with confirmation). The actual files on disk are not modified or deleted.
  • Register to Library Manager - registers the library directly into Library Manager as a signed, tracked library without needing to export and re-import a package. See Registering Libraries below.
  • Export .hxlibpkg - saves the library as a signed .hxlibpkg package and automatically registers it into Library Manager in one step. See Exporting as a Package below.

Registering Libraries

The Register to Library Manager button allows you to bring an unsigned library under full Library Manager tracking in a single click, without needing to export a .hxlibpkg package file first. This is the fastest way to convert an unmanaged library into a signed, managed library.

What Registration Does

  1. Any pending metadata edits (author, version, description, tags, etc.) are automatically saved
  2. An installed_libs.json database record is created with the same schema used by the standard import flow (library name, author, version, file list, install path, etc.)
  3. Integrity hashes are computed for all qualifying library files (.hsl, .hs_, .sub, and COM DLLs) using the same computeLibraryHashes() function used during import
  4. Public functions are extracted from .hsl files for display in the library detail modal
  5. Required dependencies (#include directives) are parsed and resolved
  6. The library is assigned to the appropriate navigation group:
    • If the author is Hamilton, the library is added to the Hamilton group
    • Otherwise, it is added to the first available custom group, or a new "Libraries" group is created if none exists
  7. The library is removed from the unsigned libraries list
  8. An audit trail entry is recorded with event type library_registered
  9. The navigation bar and library cards are refreshed
  10. A success modal is displayed confirming registration

Note: Registration does not move or copy any files on disk. The library files remain in their current location. Library Manager simply creates a tracking record that references the existing files. This means the library will no longer appear in the Unsigned group and will instead appear as a regular managed library card in its assigned group.

Tip: If you need a distributable .hxlibpkg package file (for sharing with colleagues or backing up), use the Export .hxlibpkg button instead. It performs the same registration automatically after creating the package file.

Exporting as a Package

The Export function converts an unsigned library into a proper .hxlibpkg package. This is useful for:

  • Distributing a library to colleagues who use Venus Library Manager
  • Creating a backup with metadata and integrity signing
  • Bringing an unmanaged library under full version and package management

Export Process

  1. Any pending metadata edits are automatically saved
  2. A save dialog prompts for the output file location (suggested filename: <LibraryName>.hxlibpkg)
  3. All library files (discovered + user-added) and demo method files are verified to still exist on disk - if any are missing, the export is aborted
  4. A manifest.json is generated from the current metadata fields, including the demo method file list
  5. Library files are added to the library/ directory inside the ZIP
  6. Demo method files are added to the demo_methods/ directory inside the ZIP
  7. If a .bmp icon exists, it is included in the icon/ directory
  8. The package is signed using signPackageZip() (HMAC-SHA256 integrity signature)
  9. An audit trail entry is recorded with source type "unsigned_library", including library and demo file counts
  10. The library is automatically registered into Library Manager as a signed/installed library (identical to clicking "Register to Library Manager" - see Registering Libraries above). The source package field in the database record is set to the exported package filename.

After a successful export and registration, a "Library Exported & Registered!" success modal is displayed. The modal includes a green status line confirming the library is now tracked by Library Manager. If the export succeeds but registration fails for any reason, the package is still saved and a warning is shown - you can then import the package manually.

Tip: Fill in the Author, Version, and Description fields before exporting. These fields become part of the package manifest and are displayed when the package is imported on another system.

Dependency Resolution

When the unsigned libraries feature is enabled, unsigned libraries participate in dependency resolution. If a library has an #include directive that resolves to a file belonging to an unsigned library:

  • The dependency is considered satisfied - it will not be listed as "missing"
  • A warning is shown instead, indicating the dependency is fulfilled by an unsigned (unpackaged) library
  • Dependencies on unsigned libraries are displayed with a yellow warning triangle and an "Unsigned" badge in the library detail modal
  • A summary line reading "N unsigned dependencies (not packaged)" appears in the dependencies section header

This ensures that users can see which of their dependencies come from unmanaged sources, while avoiding false-positive "missing dependency" alerts for libraries that do exist on disk.

If the feature is disabled (checkbox unchecked), unsigned libraries are excluded from dependency resolution and any dependencies on those files will appear as "missing" or "unknown" as before.

Data Model

Unsigned library records are stored in unsigned_libs.json within the user data directory. Each record has the following structure:

{
  "_id": "ulib_mylibrary",
  "library_name": "MyLibrary",
  "author": "",
  "organization": "",
  "version": "",
  "venus_compatibility": "",
  "description": "",
  "tags": [],
  "library_files": ["MyLibrary.hsl", "MyLibrary.hs_", "MyLibrary.bmp"],
  "additional_library_files": [],
  "demo_method_files": [],
  "com_register_dlls": [],
  "lib_base_path": "C:\\Program Files (x86)\\HAMILTON\\Library",
  "library_image": "MyLibrary.bmp",
  "library_image_base64": "Qk12...",
  "library_image_mime": "image/bmp",
  "scanned_date": "2026-02-24T12:00:00.000Z",
  "is_unsigned": true
}
FieldTypeDescription
_idstringDerived from the library name: ulib_ + lowercase alphanumeric name
library_namestringDisplay name derived from the primary .hsl or .smt filename
authorstringUser-editable author name (blank until set)
organizationstringUser-editable organization
versionstringUser-editable version string
venus_compatibilitystringUser-editable VENUS compatibility version
descriptionstringUser-editable description text
tagsstring[]User-editable keyword tags
library_filesstring[]Relative file paths discovered by the scanner
additional_library_filesstring[]Absolute paths to user-added library files (not originally discovered)
demo_method_filesstring[]Absolute paths to user-added demo method files
com_register_dllsstring[]Basenames of DLLs marked for COM registration via RegAsm
lib_base_pathstringAbsolute path to the Library folder root
library_imagestring | nullFilename of the icon image (if discovered)
library_image_base64string | nullBase64-encoded icon data
library_image_mimestring | nullMIME type of the icon
scanned_datestring (ISO 8601)Timestamp of the last scan that discovered this library
is_unsignedbooleanAlways true - distinguishes from installed library records

Limitations

  • No hashing or integrity checking - unsigned libraries are not hashed or validated. This feature is for convenience only.
  • No automatic file watching - files placed in the Library folder between scans are not detected in real-time. However, the Scan for new unsigned libraries each launch option can be enabled to automatically re-scan at startup, or you can click Scan Now at any time.
  • Flat grouping - all unsigned libraries are placed in the single Unsigned group. They cannot be moved to custom user groups.
  • No version history - unsigned libraries are not cached in the package store and do not support version rollback. However, once registered or exported as a package, the library gains full version management capabilities.
  • Metadata is user-supplied - unlike installed libraries, metadata fields (author, version, etc.) start blank and must be filled in manually.
  • File removal is soft - removing an unsigned library from the list only deletes the database record. The actual files on disk remain untouched. The library will reappear on the next scan unless the files are manually deleted, claimed by a proper import, or registered using the "Register to Library Manager" button.
  • Dot-prefixed files and folders are excluded - any file or directory whose name starts with . (e.g., .MyLibrary.hsl, .backup/) is treated as a hidden or temporary artifact and is completely excluded from scanning. The same applies to tilde-prefixed files. See Excluded Files and Directories for the full list of exclusion rules.
