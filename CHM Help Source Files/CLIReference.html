Overview

Venus Library Manager ships a full-featured command-line interface (cli.js) that mirrors the GUI workflows. The CLI enables scripted automation, CI/CD integration, headless testing, and batch operations on Hamilton VENUS libraries.

Prerequisites

  • Node.js must be installed (or use the NW.js nw binary from the project directory)
  • All Node.js dependencies (adm-zip, diskdb) are satisfied by the project's node_modules

Usage

cd "C:\Users\admin\Documents\GitHub\Library Manager"
node cli.js <command> [options]

Available Commands

CommandDescription
list-libsList all installed libraries
import-libImport a single .hxlibpkg file
import-archiveImport a .hxlibarch archive (multiple libraries)
export-libExport a single installed library as .hxlibpkg
export-archiveExport one or more installed libraries as .hxlibarch

create-packageCreate a .hxlibpkg package from a JSON spec file
list-versionsList cached package versions for a library
rollback-libReinstall a previously cached version of a library

helpDisplay help text

Global Options

These options are available for all commands:

FlagDescription
--db-path <dir>Override path to the user data directory. Default: read from settings.json, or <Hamilton Library>\.LibraryManager
--store-dir <dir>Override package store location. Default: C:\Program Files (x86)\HAMILTON\Library\LibraryPackages

User Data Directory Resolution

The user data directory (where installed_libs.json, groups.json, tree.json, and links.json are stored) is resolved in this priority order:

  1. --db-path flag on the command line (highest priority)
  2. userDataPath value in db/settings.json
  3. Default: C:\Program Files (x86)\HAMILTON\Library\.LibraryManager

If the directory does not exist, it is created and seeded with empty JSON collections.

Exit Codes

CodeMeaning
0Success
1 (non-zero)Failure — an error message is printed to stderr

All commands exit with code 0 on success and non-zero on failure, making them suitable for CI/CD systems that check process exit codes.

COM DLL Registration Note

Important: COM DLL registration is not performed by the CLI. The CLI records COM DLL intent in the database. Run RegAsm.exe /codebase <dll> with administrator rights manually after importing a package that requires COM registration. To deregister, use RegAsm.exe /unregister <dll>.

Automated Testing / CI Usage

The CLI is designed for use in automated test pipelines. A typical round-trip test sequence:

@REM 1. Create a package from raw sources
node cli.js create-package --spec tests\MyLib.spec.json --output tests\out\MyLib.hxlibpkg

@REM 2. Import it against a test database
node cli.js import-lib --file tests\out\MyLib.hxlibpkg ^
    --db-path tests\scratch-db ^
    --lib-dir tests\scratch-lib ^
    --met-dir tests\scratch-met

@REM 3. Verify it appears
node cli.js list-libs --db-path tests\scratch-db --json

@REM 4. Re-export the installed library
node cli.js export-lib --name "MyLib" ^
    --db-path tests\scratch-db ^
    --output tests\out\MyLib-roundtrip.hxlibpkg

@REM 5. Archive all libraries
node cli.js export-archive --all ^
    --db-path tests\scratch-db ^
    --output tests\out\archive.hxlibarch

@REM 6. Clean up
node cli.js delete-lib --name "MyLib" --db-path tests\scratch-db --yes

By using --db-path, --lib-dir, and --met-dir overrides, tests can run against isolated scratch directories without affecting production library installations.
