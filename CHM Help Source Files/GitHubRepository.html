GitHub Repository Link
Library Manager » Creating Packages » GitHub Repository Link

Overview

Library packages can optionally include the URL of their source GitHub repository. When present, this link is displayed in the library detail views so that users can navigate directly to the project's source code, issues, documentation, and releases.

The field is labelled "GitHub Repository" in both the GUI Library Packager and the Unsigned Library detail editor. The label is intentionally descriptive: it asks for the repository URL — not a personal profile, organization page, or other GitHub link.

Where the Field Appears

Authoring (Input)

  • GUI Library Packager — an optional text field between Description and Default Tags in the Package Metadata card.
  • Unsigned Library Detail Editor — an identical field in the Package Metadata card of the unsigned library modal.
  • CLI create-package --spec — the github_url key in the package spec JSON.

Viewing (Output)

  • Library Detail Modal (“View More Details”) — a GitHub section appears immediately below Description, showing the URL as a clickable link. If the library has no GitHub URL, this section is completely hidden.
  • Import Preview Modal — the same conditional section is shown during the pre-import preview so users can see the repository before confirming the import.

Data Flow

  1. Packaging: When a .hxlibpkg file is built, the URL is written into the manifest.json inside the package ZIP as the github_url field. If the field is empty, it is omitted from the manifest entirely.
  2. Importing: All import code paths (GUI single-file import, GUI archive import, and CLI import-lib / import-archive) copy manifest.github_url into the installed library database record.
  3. Exporting: When a library is re-exported (GUI or CLI export-lib / export-archive), the stored github_url is written back into the new manifest so it round-trips through export/import cycles.
  4. Viewing: The detail modal checks the github_url property of the database record and the chk_showGitHubLinks display setting. If present and the setting is enabled, the GitHub section is shown with a clickable link; otherwise, the section remains hidden (d-none).

Validation Rules

The field is optional. When left empty it is silently omitted. When a value is provided, it must pass the validateGitHubRepoUrl() validation (defined in lib/shared.js) before the package can be created or the record saved. Invalid URLs are rejected with a descriptive error message and the field is highlighted in red.

Accepted Formats

  • https://github.com/owner/repo (with or without trailing slash)
  • https://github.com/owner/repo.git
  • https://github.com/owner/repo/issues, /pulls, /tree/main, /blob/main/README.md, /releases, and other repo-scoped routes
  • Both http and https schemes are allowed
  • Non-GitHub hostnames are permitted (e.g., GitHub Enterprise, GitLab, or Gitea instances)

Rejection Criteria

The validator enforces a comprehensive set of rules to ensure the URL points to an actual repository and not a profile page, settings screen, or other non-repository GitHub resource:

RuleExample Rejected
Scheme must be http or https ftp://github.com/owner/repo, ssh://git@github.com/owner/repo
Hostname must be present https:///owner/repo
Path must have at least two non-empty segments (/{owner}/{repo}) https://github.com/octocat (owner only)
Neither segment may contain spaces or URL-encoded control characters https://github.com/my%20owner/repo
Neither segment may contain @ https://github.com/user@host/repo
Owner segment must not start or end with . or contain .. https://github.com/.hidden/repo, https://github.com/ow..ner/repo
Repo must not be empty after stripping .git https://github.com/owner/.git
First segment must not be a reserved GitHub top-level route https://github.com/settings/profile, https://github.com/login/oauth
Second segment must not be a reserved GitHub route https://github.com/owner/settings, https://github.com/owner/marketplace
Path must not match reserved non-repo prefix patterns https://github.com/users/someone/repos, https://github.com/orgs/myorg/repos
Third segment (if present) must not indicate a non-repo context https://github.com/owner/repo/organizations, https://github.com/owner/repo/search

Reserved Top-Level Routes

The following first-segment values are blocked (case-insensitive). They are known GitHub platform routes that cannot be user/organization names:

settings    organizations  orgs         users        sessions
login       logout         join         pricing      features
marketplace topics         explore      trending     collections
events      sponsors       security     site         about
contact     customer-stories  readme    codespaces   copilot
enterprises enterprise     account      notifications  inbox
watch       pulls          issues       search       apps
new         gist           gists        blog         discussions
actions     support

Manifest Schema

In the package spec JSON schema (cli-schema.json), the field is defined as:

{
  "github_url": {
    "type": "string",
    "format": "uri",
    "description": "URL to the library's GitHub repository. Optional.",
    "examples": ["https://github.com/user/venus-library"]
  }
}

Examples

GUI Library Packager

In the Package Metadata card, fill in the GitHub Repository field with the repository's web URL:

https://github.com/acme-pharma/venus-ml600-helpers

Leave the field blank to omit the link from the package.

CLI Spec File

{
  "author": "Jane Smith",
  "organization": "Acme Pharma",
  "version": "1.0.0",
  "venus_compatibility": "4.7+",
  "description": "High-level pipetting helpers for the ML600.",
  "github_url": "https://github.com/acme-pharma/venus-ml600-helpers",
  "tags": ["pipetting", "ML600"],
  "library_files": ["ML600Helpers.hsl", "ML600Helpers.hs_"]
}

Library Detail Modal

After importing a package that contains a github_url, opening View More Details shows a GitHub section below the description with the URL rendered as a clickable hyperlink. Libraries without a github_url do not display this section at all.

Display Setting — Regulated Environments

In regulated or locked-down environments, administrators may want to prevent external links from appearing in library detail views. The Settings > Display section provides a checkbox:

Show GitHub repository links in library details

When unchecked, the GitHub section is hidden from both the Library Detail Modal and the Import Preview Modal, regardless of whether a github_url value exists in the library record. The underlying data is preserved — only the display is suppressed. Re-enabling the checkbox immediately restores GitHub link visibility.

This setting is stored as chk_showGitHubLinks in the settings database. It defaults to true (enabled). See Data Model for details.
