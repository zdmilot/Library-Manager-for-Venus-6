HSL File Architecture
Library Manager » HSL File Architecture

Overview

Hamilton VENUS uses a proprietary text-based scripting language called HSL (Hamilton Standard Language). HSL source files carry a metadata footer on their last line that controls read-only behavior, tracks authorship, and provides a built-in integrity checksum. Understanding this footer is essential for verifying system library integrity.

Note: The .hs_ extension is not an inherent read-only indicator. While Hamilton conventionally uses .hs_ for protected include files, the actual read-only determination is made by the metadata footer inside the file, not the extension.

The Metadata Footer

Every HSL-type file (.hsl, .hs_, .smt) that has been saved by a Hamilton editor contains a metadata footer comment as its last line:

// $$author=NAME$$valid=V$$time=TIMESTAMP$$checksum=HEXHASH$$length=NNN$$

Footer Fields

FieldTypeDescription
authorstringWindows username or account that last saved the file
validinteger (0 or 1)Read-only / validation flag. 1 = Hamilton-validated (read-only in editors), 0 = user-modifiable
timestringTimestamp of last save (format: YYYY-MM-DD HH:MM)
checksumhex stringHamilton's CRC32-based checksum over the file body
lengthintegerInternal length indicator used by Hamilton tools

Examples

Hamilton system library (protected, read-only):

// $$author=ContainerAdministrator$$valid=1$$time=2024-06-14 18:46$$checksum=ab13e496$$length=099$$

User-created library (editable):

// $$author=walton_s$$valid=0$$time=2014-09-08 09:22$$checksum=2a469957$$length=085$$

How the valid Flag Works

The $$valid=1$$ flag is the true indicator that the Hamilton HSL Method Editor uses to determine read-only status:

  1. The editor reads the last line of the file
  2. If $$valid=1$$ is present, the file opens in read-only mode
  3. If $$valid=0$$ or no footer exists, the file opens in editable mode

Key Points

  • The Windows file system read-only attribute is NOT used by Hamilton tools
  • The .hs_ extension does NOT enforce read-only - files with .hs_ and valid=0 are editable
  • You cannot set valid=1 through the Method Editor - this flag is reserved for Hamilton's build/release process

How the checksum Field Works

The $$checksum=HEXHASH$$ contains a CRC32-based hash computed by Hamilton's tools over the file content excluding the footer line. This provides tamper detection:

  • If the file body is edited without going through Hamilton's editor, the checksum will no longer match
  • When valid=1, the combination of the valid flag and checksum provides a strong signal that the file is an unmodified system library
  • When valid=0, the checksum still provides change-detection, but the file is user-territory

Files Without a Footer

Some HSL files may lack the metadata footer. This occurs when:

  • The file was created outside of Hamilton tools (e.g., hand-written in a text editor)
  • The file is a legacy file from early VENUS versions
  • The file was generated by third-party tooling

Files without a footer are treated as editable and are not tracked in the system library baseline.

Footer Parsing (Regex Reference)

To parse the metadata footer programmatically:

// JavaScript
const regex = /\$\$author=(.+?)\$\$valid=(\d)\$\$time=(.+?)\$\$checksum=([a-f0-9]+)\$\$length=(\d+)\$\$/;
const match = lastLine.match(regex);
if (match) {
    const author   = match[1];
    const valid    = parseInt(match[2], 10);  // 0 or 1
    const time     = match[3];
    const checksum = match[4];                // hex CRC
    const length   = parseInt(match[5], 10);
}
