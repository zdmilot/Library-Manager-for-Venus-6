Integrity Verification
Venus Library Manager » Integrity Verification

Overview

Venus Library Manager provides a multi-layered integrity system to detect when library files have been modified, tampered with, or deleted. The system uses four complementary mechanisms depending on the context:

  1. Binary container HMAC - cryptographic whole-file integrity check on every .hxlibpkg and .hxlibarch file (see Binary Container Format)
  2. HMAC-SHA256 package signing - per-file integrity signature (signature.json) inside each package’s internal ZIP
  3. SHA-256 file hashing - tracks individual files in user-installed libraries after import
  4. Hamilton metadata footer verification - checks Hamilton’s built-in $$valid$$ / $$checksum$$ fields in system library HSL files

Together, these mechanisms ensure that package files cannot be casually opened or modified, that packages are intact before import, that library files remain unchanged from the time they were installed, and that system libraries match Hamilton’s original release state.

User-Installed Libraries (SHA-256 File Hashing)

For libraries imported through Venus Library Manager (via GUI or CLI), SHA-256 cryptographic hashes are computed at install time and stored in the database record's file_hashes field. On each subsequent view of the library card or detail modal, the current file hashes are recomputed and compared against the stored baseline.

How Hashing Works

The hash computation varies by file type:

File TypeHashing MethodReason
.hsl, .hs_, .subSHA-256 of all content except the last line (line endings normalised to \n)The last line contains Hamilton's metadata footer (timestamps, checksums) which changes whenever a file is re-saved in the VENUS Method Editor, even without functional changes. Excluding it prevents false positives.
.dll (COM-registered only)SHA-256 of the entire binary fileDLLs have no metadata footer, so the full content is hashed.
Other (.bmp, .ico, .chm, .stp)Not hashedBinary resource files do not contain executable logic - their modification does not affect library behavior.

When Hashes Are Computed

  • At install time: After all files are extracted from a .hxlibpkg, the function computeLibraryHashes iterates every file in the library's library_files list. Files with qualifying extensions (.hsl, .hs_, .sub) or files listed in com_register_dlls are hashed. The resulting hash map is saved to the database record.
  • On each view: When you open a library card or detail modal, verifyLibraryIntegrity recomputes the hash of each tracked file and compares it to the stored value.
  • After repair: When a library is repaired from a cached package, hashes are recomputed and the database record is updated with fresh values.

Integrity States

StateIconMeaning

Visual Indicators

Library cards on the main screen use color-coded styling to indicate integrity status at a glance:

  • Normal card styling: All files pass integrity checks.
  • Red / error border: One or more tracked files are missing or have been modified. The card also shows a red tooltip with details.
  • Yellow / warning icon: Non-critical issues such as legacy libraries without hash data.

System Libraries (Hamilton Metadata Footer)

Hamilton system libraries (the built-in libraries shipped with VENUS) use a different integrity mechanism. Instead of SHA-256 hashing, Venus Library Manager reads Hamilton's own metadata footer embedded at the end of HSL-type files. See HSL File Architecture for a detailed explanation of the footer format.

How System Library Verification Works

  1. Baseline generation: The CLI command generate-syslib-hashes scans a known-good VENUS Library folder and extracts the $$valid$$ flag, $$checksum$$, and other metadata footer fields from every HSL-type file (.hsl, .hs_, .smt). The extracted data is saved to db/system_library_hashes.json.
  2. Runtime verification: When you view a system library card or open the Verify & Repair tool, the function verifySystemLibraryIntegrity re-reads each file's footer and compares it field-by-field against the stored baseline.

What the Footer Contains

Each HSL-type file may contain a single-line metadata footer at the end with the following fields:

$$author=Hamilton$$valid=1$$time=2024-01-15 10:30:00$$checksum=a1b2c3d4e5f6$$length=12345$$
FieldDescription
$$author$$The author or tool that last saved the file (typically "Hamilton")
$$valid$$1 = file is in a validated/locked state; 0 = file has been modified or re-saved outside Hamilton's compilation
$$checksum$$A hex checksum computed by Hamilton's tools over the file content
$$time$$Timestamp of the last save
$$length$$Expected content length in bytes

Detection Criteria

ConditionResultSeverity
File no longer exists on diskFile missingError
Footer removed entirely (baseline had valid=1)Metadata footer removedError
Footer removed (baseline had valid≠1)Metadata footer removedWarning
valid changed from 1 to 0Valid flag changed - file was re-saved or unlockedError
checksum differs from baselineChecksum changed - file content was modifiedError
All fields match baselineOKPass
No baseline exists for this libraryNo integrity baseline storedWarning

What Is NOT Tracked for System Libraries

Binary resource files (.bmp, .ico, .chm, .stp, .dll) are not baselined for system libraries. Only HSL-type files with Hamilton's native metadata footer are tracked. This keeps the baseline portable and lightweight.

HSL Files Without a Footer

HSL-type files that lack a metadata footer (created outside Hamilton tools, legacy files, or third-party files) are reported during baseline generation but are not tracked, since there is no Hamilton-native integrity indicator for them.

Package Signing (HMAC-SHA256)

In addition to file-level integrity checking after installation, Venus Library Manager embeds a cryptographic signature in every .hxlibpkg package at creation time. This inner package signing provides per-file integrity verification within the package’s internal ZIP archive, complementing the binary container HMAC that protects the file as a whole.

How Package Signing Works

  1. When a .hxlibpkg is created (via the GUI Library Packager, CLI create-package, or single-library export), SHA-256 hashes are computed for every entry in the internal ZIP archive except the signature itself.
  2. The hashes are collected into a sorted map of { "entryName": "sha256hex" }.
  3. An HMAC-SHA256 is computed over the canonical JSON serialization of that map using a fixed application key.
  4. A signature.json file is added to the ZIP containing:
    • format_version - always "1.0"
    • algorithm - always "HMAC-SHA256"
    • signed_date - ISO 8601 timestamp
    • file_hashes - the per-entry SHA-256 hash map
    • hmac - the HMAC-SHA256 digest

Three-Phase Verification

When a package is imported or used for repair, the binary container is first validated (see Binary Container Format), then three inner verification phases run in sequence:

PhaseCheckFailure Meaning
1. PresenceDoes signature.json exist in the ZIP?Package is unsigned - it was created by a third party or an older version. Import proceeds with a notice; the package is not considered failed.
2. HMACRecompute HMAC from the stored file_hashes and compare to the stored hmac.signature.json itself has been tampered with.
3. ContentRecompute SHA-256 of each actual ZIP entry and compare to the signed hashes.Individual files inside the package have been modified, removed, or injected since signing.

Verification Outcomes

ResultGUI BehaviorCLI Behavior
VerifiedImport proceeds normallyImport proceeds normally
UnsignedImport proceeds with a noticeImport proceeds with a warning message
FailedUser is warned and asked to confirm or cancelImport fails unless --force is used

During archive (.hxlibarch) import, packages with failed signatures are skipped automatically.

First-Run System Library Backup

On the very first launch of Venus Library Manager (or whenever the sysLibBackupComplete setting has not been set), the application automatically creates a signed backup package for every Hamilton system library. This provides a known-good recovery point if system library files are later modified or corrupted.

What Happens at First Run

  1. During application startup, the function ensureSystemLibraryBackups checks whether the sysLibBackupComplete flag exists in the settings database.
  2. If the flag is missing or false, the function iterates through every system library defined in db/system_libraries.json.
  3. For each system library that does not already have an entry in the package store, a .hxlibpkg backup is created:
    • All discovered library files (e.g., .hsl, .hs_, .bmp, .ico, .stp) are packaged under a library/ folder in the ZIP.
    • All help files (.chm) are packaged under a help_files/ folder in the ZIP.
    • A manifest.json is created with version "system", author "Hamilton", and the tags ["system", "hamilton", "backup"]. It also includes the flag is_system_backup: true.
    • The ZIP is signed using the same HMAC-SHA256 mechanism as regular packages (via signPackageZip).
    • No demo method files are included - system libraries do not ship with demo methods.
    • If the library has a .bmp file, it is embedded in the manifest as a Base64-encoded library image.
  4. The backup package is stored in the same LibraryPackages directory used by all imported user packages, organised by library name: 
    C:\Program Files (x86)\HAMILTON\Library\LibraryPackages\
  └── HSLBarcodeReader\
      └── HSLBarcodeReader_vsystem_20260224-103000.hxlibpkg
  5. After all libraries are processed, the setting sysLibBackupComplete is set to true so the backup does not run again on subsequent launches.
  6. Progress and any errors are logged to the developer console.

Backup Contents

Each system library backup package contains:

ZIP EntryContents
manifest.jsonPackage metadata: library name, author, version ("system"), description, tags, file lists, optional Base64 image
library/All non-CHM library files: .hsl, .hs_, .bmp, .ico, .stp, .sub, etc.
help_files/All .chm help files
signature.jsonHMAC-SHA256 integrity signature covering all entries

Note: System libraries already present in the package store (e.g., from a previous installation) are skipped during backup. The first-run backup only creates packages for libraries that have no existing cached version.

Repair

When integrity errors are detected for any library (user-installed or system), Venus Library Manager can restore the library to a known-good state by re-extracting files from a cached or backup .hxlibpkg package in the package store.

How Repair Works

The repair process follows the same steps regardless of whether the library is a user-installed library or a system library:

  1. The newest cached .hxlibpkg for the library is located in the package store.
  2. The package's HMAC-SHA256 signature is verified. If the signature is present and invalid, the repair is aborted - repairing from a corrupt source would make matters worse.
  3. The package's manifest.json is read and validated.
  4. Files are extracted from the ZIP back to their original locations, overwriting the modified or corrupted files on disk:
    • library/* entries → the library's install directory
    • help_files/* entries → the library's install directory (system libraries) or library directory (user libraries)
    • demo_methods/* entries → the demo methods directory (user libraries only; system libraries have no demo methods)
  5. The UI is refreshed to reflect the restored state.

Differences Between User and System Library Repair

AspectUser Library RepairSystem Library Repair
Backup sourceCached .hxlibpkg from original importBackup .hxlibpkg created at first run
Demo methodsRe-extracted if present in the cached packageNot applicable - system libraries have no demo methods
Hash update after repairSHA-256 hashes are recomputed and saved to the databaseNot needed - system libraries use footer-based verification, not hash-based
Target directoryThe library's lib_install_path and demo_install_path from the database recordThe VENUS Library directory (typically C:\Program Files (x86)\HAMILTON\Library)

Where to Access Repair

There are two places in the GUI to repair a library:

1. Library Detail Modal

Open any library (user or system) by clicking its card. If the File Integrity section shows errors and a backup package exists in the package store, a Repair from Backup button appears below the error list. Clicking it prompts for confirmation, then restores the files. The detail modal automatically refreshes after repair.

2. Verify & Repair Tool

Open the overflow menu (⋮ button in the top-right toolbar) and select Verify & Repair. This opens a modal that scans all libraries - both system and user-installed - and displays their integrity status in a single list. See the Verify & Repair Tool section below for details.

Verify & Repair Tool

The Verify & Repair tool provides a comprehensive integrity scan and one-click repair across all libraries. Access it from the overflow menu (⋮) → Verify & Repair.

Repair Options

  • Individual Repair: Each failed library with a cached/backup package shows a Repair button. Clicking it restores that single library.
  • Batch Repair: The Repair Selected button at the bottom of the modal repairs all failed libraries in one operation. A confirmation dialog lists all libraries that will be repaired. After repair, a summary is shown with the count of successful repairs and any errors.
  • No cached package: If a library has integrity issues but no cached or backup package exists, a "No cached pkg" or "No backup" label is shown instead of a repair button. In this case, you must re-import the original package file or reinstall the library manually.

Note: The cached package's HMAC-SHA256 signature is always verified before repair. If the backup package itself is corrupted, repair is aborted to prevent restoring bad files.

Package Store

The package store is the central repository for cached and backup .hxlibpkg files. It serves both version rollback and the repair functionality. The default location is:

C:\Program Files (x86)\HAMILTON\Library\LibraryPackages

CLI Commands

  • CLI: generate-syslib-hashes - Generate system library integrity baseline from a known-good VENUS installation
  • CLI: verify-syslib-hashes - Verify system libraries against the stored baseline
  • CLI: verify-package - Verify a .hxlibpkg file's HMAC-SHA256 signature

Practical Recommendations

  • Generate a baseline early: Run generate-syslib-hashes from a freshly installed, unmodified VENUS environment to capture a clean baseline.
  • Verify periodically: Run verify-syslib-hashes or use the GUI Verify & Repair tool periodically, or as part of your instrument qualification (IQ/OQ) process.
  • Investigate TAMPERED results: Any file flagged as modified may indicate unauthorized changes to validated system libraries. Determine whether the change was intentional before proceeding.
  • Monitor user library integrity: Use the GUI integrity indicators on library cards to catch unexpected file modifications before running methods.
  • Validate received packages: Use verify-package to validate .hxlibpkg files received from external sources before importing.
  • Avoid deleting package store: The LibraryPackages directory is your recovery source. Deleting its contents removes the ability to repair or roll back libraries.
  • First-run backup: The first launch automatically backs up all system libraries. Ensure the application has write access to the LibraryPackages directory on first run.
  • Repair from the detail modal: For quick single-library fixes, open the library card and use the Repair from Backup button in the File Integrity section.
  • Batch repair: For comprehensive recovery, use the Verify & Repair tool from the overflow menu to scan everything and repair all issues at once.
