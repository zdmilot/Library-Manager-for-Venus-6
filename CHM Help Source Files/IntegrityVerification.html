Integrity Verification
Venus Library Manager » Integrity Verification

Overview

Venus Library Manager provides file integrity verification to detect when library files have been modified, tampered with, or deleted after installation. It uses two different integrity strategies depending on whether a library is user-installed or a Hamilton system library.

User-Installed Libraries (SHA-256)

For libraries imported through Venus Library Manager, SHA-256 cryptographic hashes are computed at install time and stored in the database record. On each subsequent view, the current file hashes are compared against the stored baseline.

Tracked File Types

ExtensionTracking Method
.hslSHA-256 hash (all content except the last line)
.hs_SHA-256 hash (all content except the last line)
.subSHA-256 hash (all content except the last line)
.dll (COM-registered only)SHA-256 hash (full file)

Note: Binary resource files (.bmp, .ico, .chm, .stp) are not hashed because they do not carry executable logic and their modification does not affect library behavior.

Integrity States

StateMeaning
All tracked files passEvery tracked file exists and its hash matches the stored baseline
Modified fileA file exists but its hash no longer matches — content has been changed
Missing fileA tracked file no longer exists at its expected path
Legacy / no-hash warningThe library was installed before integrity tracking was added, or hashes were not computed

System Libraries (Hamilton Metadata Footer)

Hamilton system libraries (built-in VENUS libraries) use a different integrity mechanism based on Hamilton's own metadata footer embedded in HSL files. See HSL File Architecture for a full explanation of the footer format.

How System Library Verification Works

  1. Baseline Generation: The generate-syslib-hashes command scans a known-good VENUS Library folder and extracts the $$valid$$ flag, $$checksum$$, and other metadata footer fields from every HSL-type file (.hsl, .hs_, .smt)
  2. Baseline Storage: The extracted data is saved to db/system_library_hashes.json
  3. Verification: The verify-syslib-hashes command re-reads each file's footer and compares it against the stored baseline

Detection Criteria

ConditionResult
Footer removed entirelyTAMPERED — Metadata footer removed
valid changed from 1 to 0TAMPERED — Valid flag changed (file was re-saved/unlocked)
checksum differs from baselineTAMPERED — File content was modified
File not found on diskMISSING
All fields match baselineOK

What Is NOT Tracked for System Libraries

Binary resource files (.bmp, .ico, .chm, .stp, .dll) are not baselined for system libraries. This keeps the baseline portable and lightweight — only files with Hamilton's native integrity mechanism are tracked.

HSL Files Without a Footer

HSL-type files that lack a metadata footer (created outside Hamilton tools, legacy files, or third-party files) are reported during baseline generation but are not tracked, since there is no Hamilton-native integrity indicator for them.

CLI Commands

  • CLI: generate-syslib-hashes — Generate system library baseline
  • CLI: verify-syslib-hashes — Verify system libraries against baseline

Practical Recommendations

  • Generate a system library baseline from a freshly installed, unmodified VENUS environment
  • Run verify-syslib-hashes periodically or as part of your instrument qualification process
  • Investigate any TAMPERED results — they may indicate unauthorized modifications to validated system libraries
  • For user libraries, use the GUI integrity indicators to catch unexpected file modifications before running methods
