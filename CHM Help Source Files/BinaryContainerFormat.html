Binary Container Format
Venus Library Manager » Binary Container Format

Overview

All .hxlibpkg and .hxlibarch files produced by Venus Library Manager use a custom binary container format. The container wraps an internal ZIP archive inside a proprietary binary envelope that is not readable or extractable using standard tools such as Notepad, 7-Zip, WinRAR, or Windows Explorer. This is a deliberate design decision: package files are application-internal artifacts, not general-purpose ZIP archives.

The binary container provides two key properties:

  • Opacity: The file cannot be casually browsed, edited, or extracted by end users or standard archive tools. The internal ZIP payload is scrambled so that no ZIP signature bytes (PK\x03\x04) are visible in the raw file.
  • Integrity: An HMAC-SHA256 digest covers the entire payload. Any modification - accidental or deliberate - is detected on import and the file is rejected. The importer fails closed: a corrupted or tampered container cannot be imported.

Important: Although the file extensions remain .hxlibpkg and .hxlibarch, these files are not ZIP archives. Attempting to open them with ZIP tools will fail. They can only be read by Venus Library Manager (GUI or CLI).

Container Layout

Every container file consists of a fixed-size 48-byte binary header followed by a variable-length scrambled payload:

Offset   Size     Field              Description
──────   ────     ─────              ───────────
0        8 bytes  Magic              Format identifier (see below)
8        4 bytes  Flags              Reserved for future use (currently 0x00000000)
12       4 bytes  Payload Length     Little-endian uint32: byte length of the scrambled payload
16       32 bytes HMAC-SHA256        Cryptographic digest of the scrambled payload

Immediately following the 48-byte header is the scrambled payload - the XOR-scrambled ZIP archive data. The total file size is always 48 + payload_length bytes.

Magic Bytes

The first 8 bytes of the file identify the container type. Two magic values are defined:

TypeExtensionMagic Bytes (hex)ASCII Representation
Single Library Package.hxlibpkg48 58 4C 50 4B 47 01 00HXLPKG\x01\x00
Multi-Library Archive.hxlibarch48 58 4C 41 52 43 01 00HXLARC\x01\x00

The last two bytes (\x01\x00) represent the container format version (version 1). This allows future format revisions to be detected and handled gracefully.

If a file does not begin with one of these two magic values, the importer rejects it immediately with an "Unrecognized file format" error.

XOR Scrambling

The internal ZIP archive is obscured using a fixed 32-byte XOR key that is cycled across the entire payload. This is a lightweight reversible transformation - not encryption - whose purpose is to prevent casual inspection and to eliminate recognizable ZIP signature bytes from the raw file.

How It Works

  1. The ZIP archive is serialized to a byte buffer.
  2. Each byte at position i in the buffer is XORed with key[i mod 32], where key is the 32-byte scramble key.
  3. The resulting scrambled buffer becomes the container payload.

Descrambling is identical: applying the same XOR operation a second time recovers the original ZIP data.

Design Rationale

The scramble layer is intentionally simple. Its goals are:

  • Prevent accidental extraction: Double-clicking a .hxlibpkg file will not open it in any archive manager.
  • Hide internal structure: No ZIP headers, file listings, or entry names are visible in hex editors or text viewers.
  • Complement integrity checking: The HMAC provides cryptographic tamper detection. The scramble adds obscurity to prevent casual editing attempts in the first place.

Note: The XOR scramble is not a security boundary. It is an obscurity layer that raises the bar for casual inspection. Integrity protection is provided by the HMAC-SHA256, which uses a cryptographically strong keyed hash.

HMAC-SHA256 Integrity

The 32-byte HMAC-SHA256 in the container header is computed over the scrambled payload using a fixed application signing key. This provides cryptographic integrity verification at the container level.

Packing (Write) Process

  1. The ZIP archive buffer is XOR-scrambled using the 32-byte key.
  2. An HMAC-SHA256 is computed over the scrambled buffer using the application signing key.
  3. The 48-byte header is assembled: magic + flags (4 zero bytes) + payload length (little-endian uint32) + HMAC digest.
  4. The header and scrambled payload are concatenated to form the final container file.

Unpacking (Read) Process

  1. The file must be at least 48 bytes (the header size). Files smaller than this are rejected.
  2. The first 8 bytes are compared to the expected magic value. A mismatch produces an "Unrecognized file format" error.
  3. The payload length is read from bytes 12-15 (little-endian uint32).
  4. The file must contain at least 48 + payload_length bytes. If the file is truncated, it is rejected.
  5. The HMAC-SHA256 is recomputed over the payload portion of the file (bytes[48 .. 48+payload_length]).
  6. The recomputed HMAC is compared to the stored HMAC (bytes 16-47) using a timing-safe comparison to prevent timing attacks.
  7. If the HMACs do not match, the file is rejected with a "Container integrity check failed" error. No data is returned.
  8. If the HMACs match, the payload is XOR-descrambled to recover the original ZIP buffer.

Fail-Closed Design: If any validation step fails - wrong magic, truncated file, HMAC mismatch - the importer rejects the file entirely. No partial data is extracted. This prevents importing corrupted or tampered packages.

Dual Integrity: Container + Inner Signature

Venus Library Manager provides defense-in-depth through two independent integrity layers:

LayerScopeMechanismWhat It Catches
Outer: Container HMACEntire file (all bytes after the header)HMAC-SHA256 over scrambled payloadAny byte change to the file - accidental corruption, disk errors, truncation, manual editing attempts
Inner: signature.jsonIndividual ZIP entries inside the containerPer-file SHA-256 hashes + HMAC over the hash mapIndividual file tampering, entry injection/removal within the ZIP (if the outer container were bypassed)

Both layers are verified during import. The container HMAC is checked first (whole-file integrity), and if it passes, the inner signature.json is checked (per-entry integrity). A failure at either layer prevents the import.

See Integrity Verification for details on the inner signature.json mechanism and Package Formats for the internal ZIP structure.

Nested Containers in Archives

A .hxlibarch archive file is itself a binary container. Inside its descrambled ZIP payload, each inner .hxlibpkg entry is also a full binary container - not a raw ZIP. This means every level has its own container header, HMAC, and scrambled payload:

MyBundle.hxlibarch (binary container)
  ├── Header: HXLARC magic + HMAC covering outer payload
  └── Scrambled payload → descrambles to ZIP containing:
      ├── archive_manifest.json
      ├── LibraryA.hxlibpkg (binary container)
      │   ├── Header: HXLPKG magic + HMAC covering inner payload
      │   └── Scrambled payload → descrambles to ZIP containing:
      │       ├── manifest.json
      │       ├── signature.json
      │       └── library/ ...
      ├── LibraryB.hxlibpkg (binary container)
      │   └── ... (same structure)
      └── LibraryC.hxlibpkg (binary container)
          └── ... (same structure)

During archive import, the outer container is first validated and descrambled, then each inner .hxlibpkg entry is individually validated and descrambled before its contents are installed. This provides consistent integrity verification at every level of nesting.

What Standard Tools See

Because the container format uses proprietary magic bytes and XOR-scrambled payloads, standard tools will show the following when encountering a .hxlibpkg or .hxlibarch file:

ToolResult
Notepad / text editorsBinary garbage - no readable text or file names are visible
7-Zip"Cannot open file as archive" - no ZIP signature detected
WinRAR"The archive is either in unknown format or damaged"
Windows ExplorerDoes not recognize the file as an archive; shows it as a generic file
Hex editorsShows the HXLPKG or HXLARC magic bytes, then binary data with no recognizable structure
Unix file command"data" - the magic bytes are not in the standard file-type database

Technical Implementation

The container format is implemented in the shared module (lib/shared.js) via two functions:

FunctionPurpose
packContainer(zipBuffer, magic)Takes a raw ZIP buffer and a magic identifier, and returns a complete binary container buffer ready to write to disk.
unpackContainer(containerBuffer, magic)Takes a container buffer read from disk and the expected magic, validates integrity, descrambles, and returns the original ZIP buffer. Throws on any error.

These functions are used by both the GUI (html/js/main.js) and the CLI (cli.js). The same signing key is used for the container HMAC and the inner signature.json HMAC.

Relationship to Other Formats

The binary container is the outermost layer of every package and archive file. Inside the container is a standard ZIP archive that contains the structured package contents described in Package Formats:

Layer 1:  Binary Container  (this page)
            ↓ descramble + HMAC verify
Layer 2:  ZIP Archive       (see Package Formats)
            ↓ extract entries
Layer 3:  Package Contents  (manifest.json, signature.json, library/, demo_methods/, icon/)
