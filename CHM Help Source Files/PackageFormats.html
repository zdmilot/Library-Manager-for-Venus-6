Package Formats
Library Manager for Venus 6 » Package Formats

Overview

Library Manager for Venus 6 uses two file formats for distributing and archiving Hamilton VENUS libraries: .hxlibpkg (single library package) and .hxlibarch (multi-library archive). Both formats use a custom binary container that wraps an internal ZIP archive inside a proprietary envelope. The binary container prevents the files from being opened, browsed, or extracted using standard tools such as 7-Zip, WinRAR, or Windows Explorer.

For full details on the binary container layer - including the header layout, XOR scrambling, HMAC-SHA256 integrity verification, and fail-closed design - see Binary Container Format.

This page describes the internal ZIP structure that lives inside each binary container: the manifest, signature, file layout, and metadata schema.

Single Library Package (.hxlibpkg)

A .hxlibpkg file is a binary container that encapsulates one complete library: its source files, demo methods, icon, and all metadata needed to install and manage the library. The container's internal ZIP archive has the following structure:

Internal ZIP Structure

Manifest (manifest.json)

Every .hxlibpkg must contain a manifest.json at the archive root. This file holds all metadata and the file inventory. Missing or invalid manifests cause the import to fail with an error.

Any unrecognized fields in a manifest are preserved during import, export, and repackaging to ensure lossless round-tripping across application versions.

Package Lineage

The package_lineage array provides a complete audit trail of every lifecycle event that has occurred to a package. Events are appended in chronological order and are never removed, creating an immutable history that survives export and re-import cycles. This enables full traceability of a package from its original creation through any number of export and repackage operations.

Lineage Event Fields

FieldTypeDescription
eventstringEvent type: "created", "exported", or "repackaged"
timestampstringISO 8601 timestamp of the event
app_versionstringLibrary Manager version at the time of the event
format_versionstringPackage format version at the time of the event
usernamestringWindows username of the operator
hostnamestringComputer hostname
windows_versionstringWindows OS version of the machine at event time
venus_versionstringHamilton VENUS version installed at event time

Lineage Accumulation

When a library is exported, an "exported" event is appended to the existing lineage from the database record. This means a package that was created on Machine A, exported on Machine B, imported on Machine C, and exported again on Machine D will contain four lineage entries — one for each lifecycle event. The environmental context (Windows version, VENUS version, hostname, operator) of every machine in the chain is permanently recorded.

Package Signature (signature.json)

Every .hxlibpkg created by Library Manager for Venus 6 includes a signature.json entry in the internal ZIP. The current v2.0 signature format includes both an HMAC-SHA256 integrity hash and an Ed25519 digital signature with an embedded publisher certificate. This inner signature covers every other entry in the ZIP and provides per-file integrity verification, complementing the outer container-level HMAC that covers the entire file. See Binary Container Format for how these two integrity layers work together.

v2.0 Signature Format (Current)

The current signature format used by Library Manager for Venus 6. Provides both HMAC-SHA256 tamper detection and Ed25519 cryptographic publisher identity verification.

FieldTypeDescription
signature_format_versionstring"2.0"
algorithmstring"Ed25519+HMAC-SHA256"
signed_datestringISO 8601 timestamp of when the signature was generated
file_hashesobjectSorted map of { "entryName": "sha256hex" } for every ZIP entry except signature.json
hmacstringHMAC-SHA256 computed over the canonical JSON representation of the file_hashes map
digital_signaturestringEd25519 digital signature (base64) computed over the canonical JSON representation of the file_hashes map
publisher_certificateobjectEmbedded publisher certificate containing identity and public key. See fields below.

    "library/MyVenusLibrary.hs_": "e5f6a7b8...",
    "manifest.json": "a1b2c3d4..."
  },
  "hmac": "9f8e7d6c...",
  "digital_signature": "MEUCIQD...",
  "publisher_certificate": {
    "cert_format": "hxlibpkg-publisher-cert/1.0",
    "publisher": "Jane Smith",
    "organization": "Acme Pharma",
    "public_key": "MCowBQYDK2VwAyEA...",
    "fingerprint": "a7e3f9d1c6b24e92b516a30de768f42c...",

  }
}

v1.0 HMAC-only signatures are rejected. Legacy packages created with signature format version 1.0 (HMAC-SHA256 only, no Ed25519 digital signature) are no longer accepted on import. Only v2.0 code-signed packages pass signature verification. To update a legacy package, re-create it using create-package with --sign-key.

Note: The inner signature.json provides per-file integrity checking and publisher identity verification within the ZIP. The outer binary container HMAC provides whole-file integrity checking. Together they form a defense-in-depth strategy. See Binary Container Format and Integrity Verification for details.

See Integrity Verification for details on how signing relates to file-level integrity checking, and CLI: verify-package for standalone package verification.

Multi-Library Archive (.hxlibarch)

A .hxlibarch file is a binary container whose internal ZIP contains multiple .hxlibpkg entries plus an archive-level manifest. Each inner .hxlibpkg entry is itself a full binary container (not a raw ZIP), providing consistent integrity verification at every level. See Nested Containers in Archives for the nesting structure.

Internal ZIP Structure

Archive Manifest (archive_manifest.json)

FieldTypeDescription
format_versionstringArchive format version ("2.0")
archive_typestringAlways "hxlibarch"
created_datestringISO 8601 timestamp of archive creation
app_versionstringVersion of Library Manager for Venus 6 that created the archive
windows_versionstringWindows OS version of the machine that created the archive
venus_versionstringHamilton VENUS version installed on the creating machine
library_countintegerNumber of packages in the archive
librariesstring[]List of library names included
archive_icon_base64stringBase64-encoded purple archive icon PNG
archive_icon_mimestringMIME type of the archive icon ("image/png")

In addition to the manifest fields, the archive ZIP also contains a physical icon file at icon/archive_icon.png. This purple archive icon (three stacked purple diamonds) visually identifies the file as a multi-library archive. See Visual Identity & Assets for details on how the icon is generated and stored.

File Type Reference

Library payload files can include any of the following Hamilton VENUS file types:

ExtensionDescriptionIntegrity Tracked
.hslHSL library definition (primary entry point)Yes (SHA-256 for user libs; footer for system libs)
.hs_HSL include / string table / implementationYes
.hsiHSL include (alternate convention)No
.smtStep module templateFooter tracked for system libs only
.stpStep template parametersNo
.subSub-method fileYes (SHA-256)
.dllDynamic-link library (may require COM registration)Yes (SHA-256, if COM-registered)
.bmpLibrary icon bitmapNo
.chmCompiled HTML Help fileNo

Note: During import, .chm files are automatically detected from the library files list and tracked separately in the help_files field so the GUI can provide quick-access help links.
