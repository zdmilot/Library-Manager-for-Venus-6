Version Rollback
Venus Library Manager » Version Rollback

Overview

Venus Library Manager automatically caches every imported .hxlibpkg file in a package store. This enables you to roll back any library to a previously installed version at any time, without needing to locate the original package file.

Package Store

The package store is a local directory that holds copies of all imported packages, organized by library name. The default location is:

C:\Program Files (x86)\HAMILTON\Library\LibraryPackages

You can override this location using the --store-dir CLI flag.

Storage Organization

Packages are stored in subdirectories named after the library. Each cached file uses a deterministic naming convention:

LibraryPackages/
  └── MyLibrary/
      ├── MyLibrary_v1.0.0_20260115-103000.hxlibpkg
      ├── MyLibrary_v1.1.0_20260201-142530.hxlibpkg
      └── MyLibrary_v2.0.0_20260220-091500.hxlibpkg

The filename format is: <LibraryName>_v<Version>_<YYYYMMDD-HHmmss>.hxlibpkg

Automatic Caching

Caching happens automatically during:

  • Single package import (import-lib) — the imported .hxlibpkg file is copied to the store
  • Archive import (import-archive) — each embedded .hxlibpkg is cached individually

You can disable caching for a specific import using the --no-cache flag.

Listing Cached Versions

To see all cached versions for a library:

node cli.js list-versions --name "MyLibrary"

Output includes for each cached entry:

  • Index number (for use with --index in rollback)
  • Version string (from the package manifest)
  • Author
  • Created date (from the manifest)
  • Cached date (when the file was stored)
  • File size
  • Filename

Results are sorted newest-first. Use --json for machine-readable output.

See CLI: list-versions for complete documentation.

Rolling Back

To reinstall a previously cached version of a library:

node cli.js rollback-lib --name "MyLibrary" --version "1.0.0"

Or by index (from list-versions output):

node cli.js rollback-lib --name "MyLibrary" --index 2

What Happens During Rollback

  1. The specified cached package is located in the package store
  2. Its manifest.json is read and validated
  3. The standard import process runs: files are extracted, the database record is replaced, integrity hashes are computed, and functions are parsed
  4. The current installed version is overwritten with the rolled-back version

Warning: Rolling back replaces the currently installed version. If the current version was modified on disk but never exported or cached, those modifications will be lost.

If neither --version nor --index is specified, the command lists all available versions and prompts you to choose one.

See CLI: rollback-lib for complete documentation.

Use Cases

  • Regression: A newly imported library version introduces problems — roll back to the previous working version
  • Testing: Compare behavior between different library versions by switching between them
  • Recovery: Restore a library that was accidentally modified or corrupted on disk
  • Audit: Verify that a specific historical version of a library produces expected results

Audit Trail

Every rollback operation is automatically recorded in the persistent audit trail (audit_trail.json). The entry captures the Windows username, Windows OS version, VENUS version, hostname, library name, and the version rolled back to. See Library Audit Log for details.
