Common Issues

Import fails with "manifest.json not found"

Cause: The .hxlibpkg file is corrupt or was not created correctly. Every valid package must contain a manifest.json at the root of the ZIP archive.

Solution:

  • Re-create the package using the create-package CLI command or the GUI Library Packager
  • Do not manually rename .zip files to .hxlibpkg - the internal structure must follow the package format specification

COM DLLs not registered after CLI import

Cause: COM registration is only performed automatically by the GUI. The CLI extracts files but cannot run RegAsm.exe.

Solution: Register manually from an elevated command prompt using the 32-bit RegAsm.exe:

C:\Windows\Microsoft.NET\Framework\v4.0.30319\RegAsm.exe /codebase "C:\Program Files (x86)\HAMILTON\Library\MyLib\MyComHelper.dll"

Critical: You must use the 32-bit RegAsm.exe from C:\Windows\Microsoft.NET\Framework\ (without "64"). Do NOT use the 64-bit version from C:\Windows\Microsoft.NET\Framework64\. Hamilton VENUS is a 32-bit application and cannot see COM objects registered in the 64-bit registry hive. Using the wrong version will cause "Class not registered" errors at runtime even though the registration command appeared to succeed.

Or import the same package using the GUI, which automatically selects the correct 32-bit RegAsm.exe.

COM registration succeeded but VENUS says "Class not registered"

Cause: The DLL was registered using the 64-bit RegAsm.exe (from C:\Windows\Microsoft.NET\Framework64\). The 64-bit version writes COM entries to the 64-bit registry hive, which is invisible to 32-bit applications like Hamilton VENUS.

Solution:

  1. First, unregister the incorrectly registered DLL using the 64-bit RegAsm: 
    C:\Windows\Microsoft.NET\Framework64\v4.0.30319\RegAsm.exe /unregister "path\to\MyComHelper.dll"
  2. Then re-register using the 32-bit RegAsm: 
    C:\Windows\Microsoft.NET\Framework\v4.0.30319\RegAsm.exe /codebase "path\to\MyComHelper.dll"

Note the path difference: Framework\ (32-bit, correct) vs. Framework64\ (64-bit, wrong for VENUS).

"System library - cannot delete" error

Cause: The library is a Hamilton built-in system library. These are protected from deletion to prevent breaking the VENUS installation.

Solution: System libraries cannot be deleted through Library Manager. If you need to modify a system library, work directly in the file system (not recommended).

"System library - cannot export" error

Cause: System libraries are read-only and cannot be exported as .hxlibpkg packages.

Solution: Only user-installed libraries can be exported. To share a system library, locate the files directly in the Library folder.

Library shows as "deleted" but files still exist

Cause: A soft delete was performed. The database record is marked as deleted, but library files remain on disk.

Solution:

  • To complete the removal, use delete with the --hard flag: node cli.js delete-lib --id <id> --hard
  • To re-import, use the import-lib command with the original package
  • Use rollback-lib if a cached version exists in the package store

Integrity verification reports "tampered" files

Cause: The file's content or metadata footer has changed since the baseline was generated.

Solution:

  • For user libraries: This means a .hsl, .hs_, .sub, or .dll file has been modified. Consider re-importing from the original package
  • For system libraries: The Hamilton metadata footer values ($$valid$$/$$checksum$$) differ from the baseline. This may indicate the file was opened and edited in the HSL Method Editor. Consider regenerating the baseline from a known-good installation

Database seems empty or out of sync

Cause: The application may be reading from the wrong user data directory, or the database files were corrupted.

Solution:

  • Verify that the local data directory exists at C:\Program Files (x86)\Library Manager\local\ and contains the expected JSON files
  • For CLI commands, verify the --db-path flag points to the correct directory
  • Shadow copies (._groups.json, ._links.json, etc.) in the data directory are diskdb backup files - do not delete them

Package store is missing or corrupt

Cause: The local/packages/ folder was deleted, moved, or has permission issues.

Solution:

  • The package store is located at: C:\Program Files (x86)\Library Manager\local\packages\
  • Ensure write permissions exist. Package caching is automatic during import - the next import will recreate the directory structure
  • If you need to disable caching for a CLI import, use --no-cache

list-versions shows no entries

Cause: No packages have been cached for that library name, or the library name does not exactly match (case-sensitive on the directory name).

Solution:

  • Check if the C:\Program Files (x86)\Library Manager\local\packages\<LibName>\ directory exists and contains .hxlibpkg files
  • Verify the exact library name spelling
  • Packages imported with --no-cache are not stored in the package store

CLI command not found / node not recognized

Cause: Node.js is not installed or not in the system PATH.

Solution: Install Node.js from nodejs.org and ensure it is available in your command prompt.

Archive import partially fails

Cause: One or more packages inside the .hxlibarch are invalid or corrupt.

Solution:

  • The valid packages in the archive are still imported - check the console output for which ones succeeded and which failed
  • Re-create the failing individual packages and rebuild the archive

Permission denied errors on Windows

Cause: The default Library path (C:\Program Files (x86)\HAMILTON\) requires administrator privileges.

Solution:

  • Run the GUI or CLI as Administrator
  • Or configure the Library and Methods directories to a user-writable location via the Settings panel or --lib-dir/--met-dir CLI flags

Package signature verification failed

Cause: The .hxlibpkg file has been corrupted during transfer, modified after creation, or tampered with. The integrity signature embedded in signature.json does not match the actual contents. For v2.0 packages, this means the Ed25519 digital signature or HMAC-SHA256 hash is invalid. Legacy v1.0 packages (HMAC-only) are rejected on import.

Solution:

  • Re-download or re-copy the package from the original source
  • Use node cli.js verify-package --file <path> to check the package before importing
  • CLI: Use --force to override the signature check when importing (use with caution)
  • GUI: A confirmation dialog allows you to proceed despite the failure
  • If the package was created by an older version of Library Manager, it may use the legacy v1.0 HMAC-only signature format, which is now rejected. Re-export the library using the current version to produce a v2.0 Ed25519-signed package. Truly unsigned packages (no signature.json at all) are still accepted.

Package shows as "unsigned"

Cause: The .hxlibpkg was created before package signing was added, or by a third-party tool that does not produce signature.json.

Solution:

  • This is not an error - unsigned packages import normally. Only packages with a failed signature are flagged as problems.
  • To add signing, re-export the library using the current version of Library Manager (GUI or CLI). The new package will include a signature.

Verify & Repair cannot repair a library

Cause: The library was imported with --no-cache, the cached package was deleted from the package store, or the cached package itself has a failed signature.

Solution:

  • Re-import the original .hxlibpkg using the Import workflow or import-lib --force
  • If you have the original package file, you can also manually copy it to the package store at C:\Program Files (x86)\Library Manager\local\packages\<LibName>\
  • Ensure the package store directory has correct write permissions

Unsigned library scan finds no libraries

Cause: All .hsl and .smt files in the Library folder are already claimed by a system library or an installed user library.

Solution:

  • This is normal if all libraries are properly managed. The scanner only finds files that are not tracked by any existing library record.
  • Verify the Library folder path is set correctly in Settings
  • If you recently imported libraries, their files are now "claimed" and will not appear as unsigned

Unsigned library reappears after removal

Cause: The "Remove" action only deletes the database record. The actual files remain on disk, so the next scan rediscovers them.

Solution:

  • To permanently prevent rediscovery, either delete the files from disk or import them as a proper .hxlibpkg package (which claims the files)
  • Export the unsigned library first (to preserve a packaged copy), then import the resulting .hxlibpkg

Unsigned library auto-scan on launch not running

Cause: The Scan for new unsigned libraries each launch checkbox is not checked, or the parent Include existing unsigned libraries on the front page checkbox is disabled.

Solution:

  • Both checkboxes must be enabled: first enable Include existing unsigned libraries on the front page, then enable Scan for new unsigned libraries each launch
  • When the parent checkbox is unchecked, the auto-scan setting is automatically reset to disabled
  • After enabling both settings, restart the application to verify the scan runs on launch (a spinning indicator will appear next to the Scan Now button, followed by a green checkmark when complete)

Event History shows no events

Cause: The audit_trail.json file does not exist or is empty in the user data directory.

Solution:

  • Verify the user data path is correctly configured in Settings
  • Perform any library action (import, export, delete, package build) to generate the first audit entry
  • If the file was deleted, it will be recreated automatically on the next recorded event

Starred libraries not persisting

Cause: The starred_libs setting failed to save to settings.json, possibly due to file permissions or a corrupt settings file.

Solution:

  • Check that local/settings.json is writable
  • If the file is corrupt, delete or rename it - it will be recreated with defaults on next launch

Run log cleanup not working

Cause: The Enable Run Log Cleanup setting is disabled, or the VENUS LogFiles directory cannot be found.

Solution:

  • Enable the setting in Settings and configure the days threshold
  • Ensure the VENUS LogFiles folder exists at the expected path
  • If using Archive mode, ensure the archive folder exists or is writable
