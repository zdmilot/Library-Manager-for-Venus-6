Code Signing
Library Manager » Code Signing

Overview

Code signing provides cryptographic proof that a library package was created by a specific publisher and has not been tampered with since it was signed. Library Manager uses Ed25519 digital signatures combined with HMAC-SHA256 tamper detection for a two-layer integrity system.

Code signing is optional — unsigned packages can still be imported and used. However, signed packages provide stronger assurances:

  • Publisher identity is cryptographically verified
  • File contents are guaranteed to be unchanged since signing
  • Packages from known OEM publishers display an OEM Verified badge
  • In Regulated Environment Mode, only signed packages can be imported

Key Concepts

Ed25519 Key Pair

Code signing uses an Ed25519 asymmetric key pair:

  • Private Key (.key.pem) — PEM-encoded PKCS#8 Ed25519 private key. Used to sign packages. Must be kept secret.
  • Publisher Certificate (.cert.json) — JSON file containing the public key, publisher name, organization, key fingerprint, and creation date. Distributed freely to package consumers.

Signature Format

When a package is code-signed, a signature.json file is embedded inside the package ZIP with the following structure:

{
  "signature_format_version": "2.0",
  "algorithm": "Ed25519+HMAC-SHA256",
  "signed_date": "2026-03-07T12:00:00.000Z",
  "file_hashes": {
    "manifest.json": "<sha256>",
    "library/MyLib.hsl": "<sha256>",
    ...
  },
  "hmac": "<hmac-sha256 hex>",
  "digital_signature": "<base64 Ed25519 signature>",
  "publisher_certificate": {
    "cert_format": "hxlibpkg-publisher-cert/1.0",
    "publisher": "Jane Smith",
    "organization": "Acme Pharma",
    "public_key": "<base64 raw Ed25519 public key>",
    "fingerprint": "<sha256 hex of raw public key bytes>",
    "key_id": "<first 16 chars of fingerprint>",
    "created_date": "<ISO 8601>"
  }
}

Verification checks both layers:

  1. HMAC-SHA256 — Verifies that the file_hashes object has not been altered
  2. Ed25519 signature — Verifies that the file hashes were signed by the private key matching the embedded public key
  3. Per-file SHA-256 — Every file inside the package ZIP is hashed and compared against the signed hash list

Setting Up Code Signing (GUI)

Code signing is configured in Settings → Code Signing:

Step 1: Generate or Select a Key Pair

You can either generate a new key pair or load an existing one:

Generate a New Key Pair

  1. Open Settings (gear icon)
  2. Scroll to the Code Signing section
  3. Click Generate Key Pair...
  4. Enter a publisher name (3–500 characters) and optional organization
  5. Choose an output directory for the key files
  6. Two files are saved: <publisher>.key.pem (private key) and <publisher>.cert.json (certificate)
  7. The key pair is automatically loaded into Settings

Load an Existing Key Pair

  1. In the Code Signing section, click Browse next to “Private Key (.key.pem)” and select your .key.pem file
  2. Click Browse next to “Certificate (.cert.json)” and select your .cert.json file
  3. The publisher name, organization, and key ID are displayed once both files are loaded successfully

Step 2: Sign Packages

Once a key pair is configured, a Code Sign toggle appears in the following locations:

  • Library Packager — When creating a new .hxlibpkg package
  • Export Choice Modal — When exporting a library from the detail view
  • Archive Export — When exporting a .hxlibarch archive

Toggle Code Sign on before building/exporting. The publisher name and key ID are displayed for confirmation. When enabled, the package is signed with your Ed25519 private key and the publisher certificate is embedded in the signature.json.

Clearing the Configuration

Click the Clear button in the Code Signing settings section to remove the current key pair configuration. This does not delete the key files from disk.

Setting Up Code Signing (CLI)

The CLI provides equivalent functionality for automated or scripted workflows:

Generate a Key Pair

node cli.js generate-keypair --publisher "Jane Smith" --organization "Acme Pharma" --output-dir C:\Keys

See CLI: generate-keypair for full options.

Create a Signed Package

node cli.js create-package --spec package-spec.json --sign-key C:\Keys\Jane_Smith.key.pem --output MyLibrary.hxlibpkg

See CLI: create-package for full options.

Verification on Import

When a signed package is imported (via GUI or CLI), the signature is verified automatically:

  1. The signature.json is read from the package
  2. HMAC-SHA256 integrity is checked — detects tampering with the hash list
  3. Ed25519 digital signature is verified against the embedded public key
  4. Every file in the package ZIP is re-hashed and compared against the signed hash list
  5. If any check fails, the import is rejected with a specific error message

On successful verification:

  • The publisher certificate is stored in the library’s database record
  • A ✔ Signed badge appears on the library card
  • The publisher name and key ID are displayed in the Library Detail Modal → Code Signing Certificate section
  • If the publisher name matches a known OEM keyword, an OEM Verified badge is displayed

Publisher Registry

Library Manager automatically maintains a registry of publisher names encountered during imports. Use CLI: list-publishers to view all registered publishers, or view publisher information in the Library Detail Modal.

OEM Publisher Protection

Packages that claim a restricted OEM author or organization name (e.g., Hamilton, Tecan, Roche, etc.) are subject to additional verification:

  • The package must be code-signed
  • The publisher certificate’s holder name must contain the OEM keyword
  • An author password is required to import or create packages with restricted names

This prevents unauthorized parties from publishing packages under established OEM brand names. See Package Spec Schema for the full list of restricted keywords.

Regulated Environment Mode

When Regulated Environment Mode is enabled, only signed packages can be imported. Unsigned packages are rejected at import time. This ensures full traceability and accountability in GxP-regulated laboratory environments.

Security Considerations

  • Private key security: The .key.pem file grants the ability to sign packages as you. Store it securely and never share it. If compromised, generate a new key pair.
  • Certificate distribution: The .cert.json file is public information and can be freely shared with anyone who will import your packages.
  • Algorithm: Ed25519 is a modern, high-performance elliptic curve algorithm endorsed by NIST (as part of EdDSA). It provides 128-bit security with 32-byte keys.
  • Timing-safe comparison: All cryptographic comparisons use crypto.timingSafeEqual() to prevent timing side-channel attacks.
