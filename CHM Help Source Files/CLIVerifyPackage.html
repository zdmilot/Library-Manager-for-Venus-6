CLI: verify-package
Library Manager » CLI Reference » verify-package

Description

Verifies the integrity of a .hxlibpkg or .hxlibarch file without importing it. This command validates both the outer binary container (HMAC-SHA256 over the entire file) and the inner signature.json (per-file integrity hashes and optional Ed25519 digital signature). For code-signed packages, publisher identity and OEM verification status are also reported. This is useful for validating packages before distribution, after file transfer, or in CI/CD pipelines.

Usage

node cli.js verify-package --file <path> [--json]

Options

FlagRequiredDescription
--file <path>RequiredPath to the .hxlibpkg or .hxlibarch file to verify
--jsonNoPrint machine-readable JSON output instead of human-readable text

Verification Process

  1. The .hxlibpkg file's binary container is validated: magic bytes, HMAC-SHA256 and payload length are checked. If the container is corrupted or tampered with, verification fails immediately.
  2. The container is descrambled to recover the internal ZIP archive
  3. The tool looks for a signature.json entry inside the internal ZIP
  4. If no signature.json is found, the package is reported as UNSIGNED (exit code 0)
  5. If signature.json is present:
    • SHA-256 hashes are recomputed for every non-signature entry in the ZIP
    • Each recomputed hash is compared against the stored hash in signature.json
    • The HMAC-SHA256 is recomputed from the file hash map and compared against the stored HMAC
    • If the signature includes an Ed25519 digital signature (signature_format_version: "2.0"), the Ed25519 signature is verified against the embedded publisher certificate's public key
    • If the publisher name matches a restricted OEM keyword, the package receives an OEM Verified badge
  6. If all hashes, the HMAC, and any Ed25519 signature match, the result is VALID (exit code 0)
  7. If any hash or the HMAC does not match, the result is FAILED (exit code 1)

Possible Outcomes

ResultExit CodeMeaning
VALID [Ed25519]0Ed25519 digital signature and all file hashes are valid. Publisher identity is confirmed.
VALID [Converted]0Package uses the 2.0-converted format (created from an installer by a companion tool). HMAC integrity is valid but no Ed25519 code signature is present.
UNSIGNED0No signature.json found - the package is a legacy or third-party package created before signing was implemented
FAILED1One or more file hashes do not match, the HMAC is invalid, the Ed25519 signature verification failed, or the package uses legacy HMAC-only format (v1.0) which is no longer accepted

Examples

:: Verify a package before importing
node cli.js verify-package --file MyLibrary.hxlibpkg

:: Machine-readable output for CI/CD pipelines
node cli.js verify-package --file MyLibrary.hxlibpkg --json

:: Verify a package after downloading from a network share
node cli.js verify-package --file "\\server\share\packages\LibraryA.hxlibpkg"

JSON Output Format

When --json is specified, the output is a JSON array of result objects (one per package; a single .hxlibpkg produces a one-element array):

[
  {
    "package": "MyLibrary.hxlibpkg",
    "signed": true,
    "valid": true,
    "code_signed": true,
    "publisher_cert": {
      "publisher": "Jane Smith",
      "organization": "Acme Pharma",
      "key_id": "a7e3f9d1c6b24e92",
      "fingerprint": "a7e3f9d1c6b24e92b516a30de768f42c..."
    },
    "oem_verified": false,
    "errors": [],
    "warnings": []
  }
]
FieldTypeDescription
packagestringFilename (or entry name for archives) of the package
signedbooleantrue if a signature.json was found
validbooleantrue if all hashes and the HMAC matched
code_signedbooleantrue if the package has an Ed25519 digital signature (v2.0 format)
publisher_certobject|nullPublisher certificate details (publisher name, organization, key_id, fingerprint) or null if not code-signed
oem_verifiedbooleantrue if the publisher name matches a restricted OEM keyword, confirming OEM origin
errorsstring[]Error messages (e.g., hash mismatches, read failures)
warningsstring[]Non-fatal warnings

Human-Readable Output

When --json is not specified, each package is printed on a single status line followed by publisher details (if code-signed):

MyLibrary.hxlibpkg: VALID [Ed25519]
  Publisher  : Jane Smith (Acme Pharma)
  Key ID     : a7e3f9d1c6b24e92
  Fingerprint: a7e3f9d1c6b24e92b516a30de768f42c...
  OEM Verified: NO

Result: ALL PACKAGES VERIFIED (code signed)

Unsigned packages display as:

LegacyLib.hxlibpkg: UNSIGNED

Result: NO SIGNED PACKAGES FOUND

Signature Format

The signature.json entry inside a signed .hxlibpkg contains:

v2.0 Format (Ed25519 Code-Signed)

Current format used by Library Manager. Includes both HMAC-SHA256 integrity and Ed25519 digital signature with publisher identity.

FieldTypeDescription
signature_format_versionstring"2.0"
algorithmstring"Ed25519+HMAC-SHA256"
signed_datestringISO 8601 timestamp of when the signature was generated
file_hashesobjectMap of { "entryName": "sha256hex", ... } for every entry in the ZIP except signature.json itself
hmacstringHMAC-SHA256 computed over the canonical JSON representation of the file_hashes map
digital_signaturestringEd25519 digital signature (base64) computed over the HMAC value
publisher_certificateobjectEmbedded publisher certificate containing: publisher, organization, public_key (base64 raw Ed25519), fingerprint (SHA-256 hex), key_id (first 16 hex chars of fingerprint)

v1.0 Format (HMAC-Only, Legacy)

Older format with HMAC integrity but no cryptographic publisher identity. Packages with this format are reported as FAILED because legacy HMAC-only signatures (v1.0) are no longer accepted. Only Ed25519 code-signed packages (v2.0) pass verification.

FieldTypeDescription
signature_format_versionstring"1.0"
algorithmstring"HMAC-SHA256"
signed_datestringISO 8601 timestamp of when the signature was generated
file_hashesobjectMap of { "entryName": "sha256hex", ... } for every ZIP entry except signature.json
hmacstringHMAC-SHA256 computed over the canonical JSON representation of the file_hashes map
