CLI: create-package
Library Manager » CLI Reference » create-package

Description

Creates a .hxlibpkg from library source files and metadata defined in a JSON specification file. This is the CLI equivalent of the GUI Library Packager, ideal for automated builds and CI/CD pipelines. The resulting package is wrapped in a binary container with HMAC-SHA256 integrity protection. Optionally, the package can be code-signed with an Ed25519 publisher certificate for cryptographic origin verification.

Usage

node cli.js create-package --spec <spec.json> --output <output.hxlibpkg> [--sign-key <key.pem>]

Options

FlagRequiredDescription
--spec <path>RequiredPath to the JSON specification file (see Package Spec JSON Schema)
--output <path>RequiredOutput .hxlibpkg file path
--sign-key <path>OptionalPath to an Ed25519 private key PEM file for code signing. When provided, the package receives both HMAC and Ed25519 digital signatures. The matching .cert.json is auto-detected from the key filename unless --sign-cert is specified.
--sign-cert <path>OptionalPath to the publisher .cert.json certificate. Only needed if the certificate filename does not match the key filename. Requires --sign-key.
--author-password <pw>OptionalRequired when the author is a restricted OEM or laboratory automation company name. Authorizes use of the restricted author name.

Specification File

The spec file is a JSON document that defines all metadata and which files to bundle. It is validated against cli-schema.json. All file paths are resolved relative to the directory containing the spec file.

Required Fields

FieldTypeDescription
authorstringLibrary author name
versionstringLibrary version string (e.g., "1.0.0")
venus_compatibilitystringVENUS version compatibility
descriptionstringHuman-readable description
library_filesstring[]Paths to library payload files (at least one required)

Optional Fields

FieldTypeDescription
library_namestringOverride library name. Auto-detected from .hsl/.hs_/.smt if omitted
organizationstringAuthor's organization
tagsstring[]Searchable keyword tags. Over 60 tag keywords are reserved (case-insensitive) and will cause the command to fail with an error, including OEM and laboratory automation company names. See Package Spec JSON Schema for the complete reserved list.
library_imagestringPath to icon/image file. Falls back to matching .bmp if omitted
demo_method_filesstring[]Paths to demo/example method files
help_filesstring[]Basenames of .chm help files
com_register_dllsstring[]Basenames of DLLs requiring COM registration
github_urlstringURL to the library's GitHub repository. Must point to a valid /{owner}/{repo} path. See GitHub Repository Link for details.
labware_filesstring[]Paths to labware definition files (.rck, .ctr, .tml, .dck, .lay). Installed under the Hamilton Labware directory.
install_to_library_rootbooleanWhen true, library files are installed directly into the Library root directory instead of a named subdirectory. Equivalent to --no-subdir at import time.
installer_executablestringPath to an optional standalone installer .exe for this library. Embedded in the package under installer/ and extracted to the installer store on import.
installer_infoobjectOptional metadata about the bundled installer. Supports description, source_url, and authenticode_signer properties.

Library Name Auto-Detection

If library_name is not specified in the spec, it is auto-detected using this priority:

  1. Basename of the first .hsl file (without extension)
  2. Basename of the first .hsi file (without extension)
  3. Basename of the first .hs_ file (without extension)
  4. Basename of the first .smt file (without extension)
  5. Parent directory name of the first library file

Image Handling

If library_image is specified and the file exists, it is embedded as the library icon (base64-encoded in the manifest and packed into the icon/ directory).

If no image is specified, the packager looks for a .bmp file among the library files whose basename (lowercase) matches the library name. If found, it is used as the icon.

Examples

:: Basic package creation
node cli.js create-package --spec MyLib.spec.json --output MyLib.hxlibpkg

:: Output to a subdirectory
node cli.js create-package --spec specs/proj.json --output dist/proj.hxlibpkg

:: Code-signed package (Ed25519)
node cli.js create-package --spec MyLib.spec.json --output MyLib.hxlibpkg --sign-key publisher.key.pem

:: Code-signed with explicit certificate path
node cli.js create-package --spec MyLib.spec.json --output MyLib.hxlibpkg --sign-key keys/publisher.key.pem --sign-cert certs/publisher.cert.json

Validation

The command validates before building:

  • author is required
  • author length must be between 3 and 500 characters; only alphanumeric characters, spaces, dashes, and underscores are allowed
  • organization length (if provided) must be between 3 and 500 characters; only alphanumeric characters, spaces, dashes, and underscores are allowed
  • version is required
  • venus_compatibility is required
  • description is required
  • library_files must contain at least one entry
  • All referenced library files must exist on disk
  • All referenced help files must exist on disk
  • Restricted OEM author or organization names require --author-password
  • Tags must not contain any of the 60+ reserved keywords (case-insensitive substring match). See Package Spec JSON Schema for the full reserved list.
  • Library name (explicit or auto-detected) must be valid (only alphanumeric characters, spaces, dashes, and underscores are allowed)

If any validation fails, the command prints a descriptive error and exits.

Auto-Generated Manifest Fields

The following fields are automatically added to the generated manifest.json and are not part of the spec file:

  • format_version — the current package format version ("2.0")
  • app_version — the version of Library Manager
  • windows_version — the Windows OS version of the build machine
  • venus_version — the Hamilton VENUS version installed at build time
  • package_lineage — an initial "created" event with operator and environment details
  • created_date — ISO 8601 timestamp of package creation

Any additional (unrecognized) properties in the spec file are preserved in the output manifest for forward compatibility.
