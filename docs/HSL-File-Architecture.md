# Hamilton HSL File Architecture — Read-Only Flags and Integrity Checking

> **Audience:** VENUS library developers, Venus Library Manager contributors, and anyone maintaining Hamilton HSL files outside the official Hamilton tools.

---

## Overview

Hamilton VENUS uses a proprietary text-based scripting language called **HSL** (Hamilton Standard Language). HSL source files carry a **metadata footer** on their last line that controls read-only behavior, tracks authorship, and provides a built-in integrity checksum. Understanding this footer is essential for programmatic integrity verification of system libraries.

---

## File Extensions

| Extension | Description | Typical Use |
|-----------|-------------|-------------|
| `.hsl` | HSL library definition | Primary library entry point; always present for a library |
| `.hs_` | HSL include / string table / implementation | Sub-module, localization strings, or private implementation |
| `.hsi` | HSL include (alternate convention) | Sometimes used for internal includes |
| `.smt` | Step module template | Step definition files used by the Method Editor |
| `.stp` | Step template parameters | Companion to `.hsl`/`.hs_`, defines step UI bindings |
| `.sub` | Sub-method file | Reusable sub-method definitions |

**Important:** The `.hs_` extension is *not* an inherent read-only indicator on its own. While Hamilton conventionally uses `.hs_` for protected include files, the actual read-only determination is made by the **metadata footer** inside the file, not the extension.

---

## The Metadata Footer

Every HSL-type file (`.hsl`, `.hs_`, `.smt`) that has been saved by a Hamilton editor contains a metadata footer comment as its **last line**. This is the authoritative source of read-only and integrity state.

### Format

```
// $$author=NAME$$valid=V$$time=TIMESTAMP$$checksum=HEXHASH$$length=NNN$$
```

### Fields

| Field | Type | Description |
|-------|------|-------------|
| `author` | string | Windows username or account that last saved the file |
| `valid` | integer (`0` or `1`) | **Read-only / validation flag.** `1` = Hamilton-validated (read-only in editors), `0` = user-modifiable |
| `time` | string | Timestamp of last save (format: `YYYY-MM-DD HH:MM`) |
| `checksum` | hex string | Hamilton's own CRC32-based checksum computed over the file body |
| `length` | integer | Internal length indicator used by Hamilton tools |

### Examples

**Hamilton system library (protected, read-only):**
```
// $$author=ContainerAdministrator$$valid=1$$time=2024-06-14 18:46$$checksum=ab13e496$$length=099$$
```

**User-created library (editable):**
```
// $$author=walton_s$$valid=0$$time=2014-09-08 09:22$$checksum=2a469957$$length=085$$
```

**Third-party library (editable):**
```
// $$author=Matt$$valid=0$$time=2015-07-13 17:23$$checksum=7706ab1e$$length=083$$
```

---

## How the `valid` Flag Works

The `$$valid=1$$` flag is the **true indicator** that the Hamilton HSL Method Editor uses to determine read-only status. When a file is opened:

1. The editor reads the last line of the file
2. If `$$valid=1$$` is present, the file opens in **read-only mode** (as shown in the title bar: "Read-Only")
3. If `$$valid=0$$` or no footer exists, the file opens in **editable mode**

### Key Points

- **The Windows file system read-only attribute is NOT used.** A file can be writable on disk but still appear read-only in the Hamilton editor.
- **The `.hs_` extension does NOT enforce read-only.** It is a naming convention, but files with `.hs_` and `valid=0` are editable.
- **The `checksum` field is Hamilton's own integrity check.** It is a CRC computed by the Hamilton tools over the file body (everything above the footer line). If the file content is modified without updating the checksum, Hamilton tools will detect the mismatch.
- **The `length` field** on Hamilton-validated files is typically `099`. User-saved files have varying values (often `080`–`090`).

---

## How the `checksum` Field Works

The `$$checksum=HEXHASH$$` field contains a hash (appears to be CRC32-based) computed by Hamilton's tools over the file content *excluding* the footer line itself. This provides tamper detection:

- If a user edits the file body without going through Hamilton's editor, the checksum will no longer match
- When `valid=1`, the combination of the valid flag and checksum provides a strong signal that the file is an unmodified Hamilton system library
- When `valid=0`, the checksum still provides a change-detection mechanism, but the file is considered user-territory

---

## Files Without a Footer

Some HSL-type files may lack the metadata footer entirely. This occurs when:

- The file was created outside of Hamilton tools (e.g., hand-written in a text editor)
- The file is a very old legacy file from early VENUS versions
- The file was generated by third-party tooling

Files without a footer are treated as **editable** by Hamilton tools. These files are not tracked in the system library baseline since there is no Hamilton-native integrity indicator for them.

---

## Integrity Verification Strategy

The Venus Library Manager uses Hamilton's built-in metadata footer as the **sole** integrity mechanism for system libraries. Only HSL-type files with footers are tracked — binary resource files (`.bmp`, `.ico`, `.chm`, `.stp`, `.dll`) are **not** baselined since they lack Hamilton's footer and hashing them would add unnecessary weight to the portable baseline.

### For HSL-type files (`.hsl`, `.hs_`, `.smt`)

Uses Hamilton's built-in metadata footer:

1. **Baseline capture:** When generating a baseline from a known-good VENUS install, the `$$valid$$`, `$$checksum$$`, `$$author$$`, `$$time$$`, and `$$length$$` fields are extracted and stored.
2. **Verification:** On each check, the current file's footer is parsed and compared against the stored baseline:
   - If `valid` changed from `1` to `0`: the file was re-saved/unlocked → **TAMPERED**
   - If `checksum` changed: the file content was modified → **TAMPERED**
   - If the footer was removed entirely: **TAMPERED**
   - If all fields match: **OK**

### For user-installed libraries (non-system)

Uses SHA-256 hashing for all tracked file types (`.hsl`, `.hs_`, `.sub`, `.dll`). User libraries do not have Hamilton's `valid=1` protection, so external hashing is the only integrity mechanism.

---

## Practical Implications

### For library developers

- When you save an HSL file in the Method Editor, it will write `valid=0` with your username and a fresh checksum
- You cannot set `valid=1` through the editor — this flag is reserved for Hamilton's build/release process
- The checksum is automatically updated on every save through Hamilton tools
- Manual edits in a text editor will NOT update the checksum, causing a mismatch detectable by integrity tools

### For system administrators

- A file with `valid=1` that suddenly shows `valid=0` has been re-saved outside Hamilton's release process
- A missing or altered checksum on a `valid=1` file indicates tampering
- The Venus Library Manager's `verify-syslib-hashes` CLI command checks all of these conditions automatically

### For the Venus Library Manager

- System library integrity is verified using Hamilton's own protection mechanism — the same one their editor uses
- Only HSL-type files with metadata footers are tracked; binary resource files are not baselined
- SHA-256 hashing is used exclusively for user-installed libraries (which lack `valid=1`)

## Reference: Metadata Footer Regex

To parse the footer programmatically:

```javascript
// JavaScript
const regex = /\$\$author=(.+?)\$\$valid=(\d)\$\$time=(.+?)\$\$checksum=([a-f0-9]+)\$\$length=(\d+)\$\$/;
const match = lastLine.match(regex);
if (match) {
    const author   = match[1];
    const valid    = parseInt(match[2], 10);  // 0 or 1
    const time     = match[3];
    const checksum = match[4];                // hex CRC
    const length   = parseInt(match[5], 10);
}
```

```python
# Python
import re
pattern = r'\$\$author=(.+?)\$\$valid=(\d)\$\$time=(.+?)\$\$checksum=([a-f0-9]+)\$\$length=(\d+)\$\$'
m = re.search(pattern, last_line)
if m:
    author, valid, time, checksum, length = m.groups()
    valid = int(valid)  # 0 or 1
```

## Summary Table

| Indicator | Controls Read-Only? | Reliable for Integrity? | Notes |
|-----------|:-------------------:|:-----------------------:|-------|
| `.hs_` extension | No | No | Naming convention only |
| `$$valid=1$$` | **Yes** | **Yes** | The authoritative read-only flag |
| `$$checksum=...$$` | No | **Yes** | Detects content changes even on `valid=0` files |
| Windows read-only attribute | No | No | Not used by Hamilton tools |
| File permissions (ACL) | Partially | No | OS-level, not HSL-level |
